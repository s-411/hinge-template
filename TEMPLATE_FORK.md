# TEMPLATE_FORK.md

> **For agents starting from this codebase as a template, not building Stage 0 from scratch.**

This repo is a reusable Expo template with ~35 fully-designed screens, a tokenised theme system, and runtime palette/font swap. The shipped demo content is a dating app called "Connect". Each fork keeps the design + tokenisation identical and adapts:

- **Content** — copy, sample profile data, paywall benefit lists, prompt library
- **Connections** — which screens link to which (router graph in [app/_layout.tsx](app/_layout.tsx))

Forks should NOT touch [theme/tokens.ts](theme/tokens.ts), the `useMemo([role])` styling pattern, or remove the `<KeyboardHideButton />` mount at the app root.

---

## When to use this guide

If your repo is a clone of this template and you're about to build a different app on top of it, follow this guide instead of the Stage-0 path in [PROCESS_GUIDE.md](PROCESS_GUIDE.md). The process pipeline still applies — you just skip stages 2–8 because the UI is already built.

If you're starting a brand-new repo with `npx create-expo-app`, use [START_NEW.md](START_NEW.md) instead.

## Per-fork checklist

Work top to bottom. Don't skip — every line is something that genuinely changes per fork.

### 1. Identity in `app.json`

| Key | Current | Change to |
|---|---|---|
| `expo.name` | `"Theme Picker"` | Your App Store display name (max 30 chars) |
| `expo.slug` | `"hinge-template"` | kebab-case unique slug (e.g. `acme-app`) |
| `expo.scheme` | `"connect"` | URL scheme for deep links (single word; usually slug minus `-app`) |
| `expo.ios.bundleIdentifier` | `"com.astrum.themepicker"` | New unique reverse-DNS bundle ID. **Once reserved in ASC, cannot be reused.** |
| `expo.extra.eas.projectId` | (existing UUID) | Generated by `eas init` (next step) |
| `expo.splash.backgroundColor` | `"#f6f1ea"` | Must match `colors.bg` in [theme/tokens.ts](theme/tokens.ts) for current light mode |
| `expo.android.adaptiveIcon.backgroundColor` | `"#f6f1ea"` | Same as splash bg |
| `expo.owner` | `"s-411"` | Stays the same across all your forks (your Expo account) |

### 2. Identity in `package.json`

| Key | Current | Change to |
|---|---|---|
| `name` | `"hinge-template"` | Same value as `expo.slug` |

### 3. EAS project init

The `eas.projectId` from step 1 must be a NEW EAS project. Run:

```bash
eas init
```

Confirm the new project is created under your `expo.owner` account. This rewrites `expo.extra.eas.projectId` automatically.

### 4. ASC App ID (set later, at Stage 11)

| File | Key | Change to |
|---|---|---|
| `eas.json` | `submit.production.ios.ascAppId` | The 10-digit ID Apple assigns when you reserve the bundle ID in App Store Connect |

`appleId`, `appleTeamId` stay the same across all your forks — they identify your Apple developer account.

### 5. Brand strings — one file

All visible app-name copy lives in [lib/brand.ts](lib/brand.ts). Edit:

```ts
export const brand = {
  name: 'Acme',
  tagline: 'Your fork-specific tagline.',
  memberLabel: 'Acme Member',
  memberLabelLower: 'Acme member',
  plusName: 'Acme+',          // or rename, e.g. 'Acme Premium'
  proName: 'Acme Pro',
  accountName: 'Acme account',
} as const;
```

After editing, every `brand.*` consumer across the app rebrands automatically. Do NOT add brand strings inline in screens — extend [lib/brand.ts](lib/brand.ts).

### 6. Demo sample data — one file

All sample profile names, ages, locations, prompt answers, paywall benefit lists live in [lib/fixtures.ts](lib/fixtures.ts). Edit each section:

- `demoProfiles.feed` — name + prompt shown on Discover (main feed)
- `demoProfiles.detail` — default profile shown at `/profile/demo` (name, age, height, location, facts list, sample prompt)
- `demoProfiles.self` — self profile name (logged-in persona)
- `demoProfiles.modal` — modal variant name
- `paywallBenefits.preferred|plus|pro` — benefit lists for each tier
- `preferredPlans`, `tierPlans.plus|pro` — pricing data (forks should swap to real RevenueCat / StoreKit pricing)
- `whatWorksCards` — 4-card editorial grid on the "What Works" modal

The `kind` strings in `paywallBenefits.plus` and `swatch` strings in `paywallBenefits.pro` are coupled to the icon switches inside [components/PaywallBody.tsx](components/PaywallBody.tsx) — keep them in sync if you add new icon kinds.

### 7. Prompt library — one file (heavily domain-specific)

[lib/prompts.ts](lib/prompts.ts) holds the question bank rendered on `/prompts/library`. The shipped questions are dating prompts; rewrite them for your vertical:

- `promptCategories` — array of category names (first one gets the "NEW" badge in the UI)
- `promptsByCategory` — `Record<Category, string[]>` mapping each category to its questions

For non-dating verticals, replace wholesale.

### 8. App icon and splash assets

Replace these four files in [assets/](assets/):

- `icon.png` (1024×1024)
- `adaptive-icon.png` (Android adaptive)
- `splash-icon.png`
- `favicon.png`

The `assets/fonts/` folder is generic — keep it unchanged.

### 9. Decide which orphan screens to keep or remove

The template ships 35 screens; some are orphaned (no incoming nav) and exposed only via the dev gallery on the **Standouts** tab ([app/(tabs)/standouts.tsx](app/(tabs)/standouts.tsx)). They are intentionally a "screen library" you can wire into your fork's flow.

**Currently orphaned (only reachable via Standouts gallery):** `/discover/photo`, `/profile/edit`, `/settings`, `/backgrounds`, `/profile/[id]`, `/matches`, plus all paywall and modal screens that aren't in the main onboarding → tabs flow.

For each, choose:
- **Wire it in** — add navigation from your fork's flow (push from a button, register as a modal, etc.)
- **Remove it** — delete the screen file AND its `<Stack.Screen name="..."/>` entry in [app/_layout.tsx](app/_layout.tsx) AND its row in [app/(tabs)/standouts.tsx](app/(tabs)/standouts.tsx)
- **Leave it** — fine for v1 if you're shipping the demo screen library as part of your fork

### 10. Update `CLAUDE.md` for your fork

Edit the "This app" section at the top with your fork's:
- Display name + slug
- New bundle ID, ASC App ID, EAS project ID
- New current stage (typically Stage 9 for forks, since 2–8 are already done)
- Your fork's purpose statement

## Process pipeline mapping for forks

[PROCESS_GUIDE.md](PROCESS_GUIDE.md) defines 16 stages. For a template fork:

| Stage | Action |
|---|---|
| 0. Triage | **Do** — fill in [BUNDLE_IDS.md](BUNDLE_IDS.md), confirm fork direction |
| 1. Repo spin-up | **Do** — clone this repo, run steps 1–8 above |
| 2–7. UI build | **Skip — already done.** Mark complete in [CLAUDE.md](CLAUDE.md) stage tracker. |
| 8a. Legal URLs | **Do** — fill privacy/terms templates, host them |
| 9. Analytics + Sentry | **Do** — see [analytics-posthog](.) and [sentry-setup](.) skills |
| 10. Self-check gate | **Do** |
| 11. App Store audit | **Do** — at this point reserve bundle ID in ASC and capture `ascAppId` (step 4 above) |
| 12. EAS build | **Do** |
| 13. TestFlight + smoke test | **Do** — see [TESTFLIGHT.md](TESTFLIGHT.md) |
| 14. Screenshots | **Do** — see [SCREENSHOT_WORKFLOW.md](SCREENSHOT_WORKFLOW.md) |
| 15. Metadata generation | **Do** |
| 8b. Landing page | **Do** — see [APP_LANDING_PAGE.md](APP_LANDING_PAGE.md) |
| 16. ASC submission | **Do** |

A typical fork from clone to TestFlight is roughly 1 day of agent work, mostly in steps 1–9 above plus stages 9–13.

## Rules — what's allowed to change

### ✅ Allowed

- Content in [lib/brand.ts](lib/brand.ts), [lib/fixtures.ts](lib/fixtures.ts), [lib/prompts.ts](lib/prompts.ts)
- Sample copy directly inline in screens (taglines, paywall headlines, button labels)
- The router graph in [app/_layout.tsx](app/_layout.tsx) — adding/removing/rewiring `<Stack.Screen>` entries
- Adding new screens that follow existing patterns (see "Adding a new screen" in [theme/README.md](theme/README.md))
- Removing orphan screens
- Editing `expo.name`, `expo.slug`, etc. per the checklist above
- Adding new entries to the `space.*` / `radius.*` / `shadow.*` scales in [theme/tokens.ts](theme/tokens.ts) when an existing entry doesn't fit (rare)

### ❌ Not allowed without explicit user sign-off

- Inline hex / rgba / font-family / fontSize values in screens — use tokens
- Module-scope `StyleSheet.create` for themed styles — must be inside `useMemo([role])`
- Changing the `colors.*` primitives in [theme/tokens.ts](theme/tokens.ts) (the user changes hue via the Settings screen at runtime, not by editing tokens)
- Removing `<KeyboardHideButton />` from [app/_layout.tsx](app/_layout.tsx)
- Hardcoding `connect.*` AsyncStorage keys (the namespace is auto-derived from `expo.slug` — see [theme/ThemeProvider.tsx](theme/ThemeProvider.tsx))
- Bumping `ios.buildNumber` in [app.json](app.json) (EAS manages it remotely; see [eas.json](eas.json))

## Verification

After your fork is set up but before Stage 9:

1. **Identity sweep clean.** `grep -ri "connect" app/ components/ lib/` should show no matches outside `lib/brand.ts` (or whatever values you put there).
2. **AsyncStorage namespace.** Boot the app in the simulator. The theme prefs persist under `<your-slug>.themeMode`, not `connect.themeMode`. Verify by inspecting AsyncStorage from a debug shell or adding a one-line `console.log(await AsyncStorage.getAllKeys())`.
3. **Brand swap proof.** Edit `brand.name` — every screen that renders "Connect Member" / "Connect+" / "Connect Pro" / "Connect account" / the Standouts intro copy reflects the new name.
4. **Demo profile swap proof.** Change `demoProfiles.detail.defaultName` from `'Bridget'` to anything. Visit `/profile/demo`. Header reflects new name.
5. **TypeScript clean.** `npx tsc --noEmit` shows only pre-existing errors (if any) in `theme/tokens.ts` — not in your changes.
6. **Visual regression.** Compare iOS simulator screenshots before/after on Welcome, all 3 tabs, one paywall, one modal. Pixel-identical except brand strings.

If verification passes, mark the template-fork "stages 2–8" complete in [CLAUDE.md](CLAUDE.md) and proceed to Stage 9.
